Crafting a Responsive Navbar That Actually Works

Japhet Sunday · June 30, 2026 · 7 min read

Navigation is one of those components that every tutorial shows wrong — a hamburger that barely opens, no keyboard support, no scroll-awareness, and active links that never actually update. Let's build one that handles all of it.

The HTML Foundation

Start with a <header> containing a logo link, a toggle button, and a <nav>. The toggle button needs aria-expanded to communicate open/closed state to screen readers. The nav itself should have an aria-label of "Main navigation" to distinguish it from other nav landmarks on the page. 

<header class="site-header" data-site-header>
  <a href="/" class="site-logo">My Site</a>

  <button
    class="nav-toggle"
    data-nav-toggle
    aria-expanded="false"
    aria-label="Toggle navigation"
  >
    <span></span>
    <span></span>
    <span></span>
  </button>

  <nav class="site-nav" data-nav-menu aria-label="Main navigation">
    <ul>
      <li><a href="#about">About</a></li>
      <li><a href="#projects">Projects</a></li>
    </ul>
  </nav>
</header>

The Hamburger Toggle

Three empty <span> elements styled as bars are the hamburger icon. CSS transforms animate them into an × when aria-expanded="true" is set. Using an attribute selector keeps the animation tied to the actual accessible state — you can't have one without the other. 

/* Default: three bars */
.nav-toggle span {
  display: block;
  width: 22px;
  height: 2px;
  background: currentColor;
  transition: transform 0.25s, opacity 0.25s;
}

/* Open state: top and bottom bars form an × */
.nav-toggle[aria-expanded="true"] span:first-child {
  transform: translateY(8px) rotate(45deg);
}
.nav-toggle[aria-expanded="true"] span:nth-child(2) {
  opacity: 0;
}
.nav-toggle[aria-expanded="true"] span:last-child {
  transform: translateY(-8px) rotate(-45deg);
}

Scroll-Aware Behaviour

A header that stays identical while the user scrolls is a missed opportunity. Two scroll states are worth implementing: scrolled (adds a backdrop blur and shadow once the user passes 60px) and hidden (slides the header up when scrolling down past the fold, reveals it when scrolling back up). The hidden state saves vertical space on mobile without removing navigation permanently.

Active Link Tracking with IntersectionObserver

The classic approach — comparing window.scrollY to section offsets on every scroll event — is both imprecise and expensive. IntersectionObserver is more elegant: observe each section, and when one enters the viewport, mark its corresponding nav link as active. 

const sections = document.querySelectorAll('section[id]');
const navLinks = document.querySelectorAll('.site-nav a[href^="#"]');

const observer = new IntersectionObserver(entries => {
  entries.forEach(entry => {
    if (!entry.isIntersecting) return;
    navLinks.forEach(link => {
      link.classList.toggle(
        'is-active',
        link.getAttribute('href') === `#${entry.target.id}`
      );
      link.setAttribute('aria-current',
        link.getAttribute('href') === `#${entry.target.id}` ? 'true' : 'false'
      );
    });
  });
}, { rootMargin: '-40% 0px -55% 0px' });

sections.forEach(section => observer.observe(section));

The rootMargin shrinks the "active" detection zone to the middle 5% of the viewport, so a section is only marked active when it's clearly the dominant content on screen — not just when its top edge crosses the viewport.

Keyboard & Focus Concerns

When the mobile menu closes, return focus to the toggle button so keyboard users aren't stranded. If the menu contains many links, trap focus inside it while it's open — Tab should cycle through the menu items, not the content behind the overlay.

The Body Scroll Lock

On mobile, add overflow: hidden to <body> while the nav is open. Without this, the page scrolls behind the overlay — disorienting and easy to miss in desktop testing.
